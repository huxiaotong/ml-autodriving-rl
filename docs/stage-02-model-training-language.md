# 第 2 阶段：理解不同模型训练的基本语言

目标：不要一上来只学 RL 的 Agent、Reward、Policy，而是先理解自动驾驶系统里不同模型“怎么被训练、怎么被评估、怎么被部署”。RL 是其中一种训练语言，不是唯一语言。

这一阶段你要建立一个总认识：

> 自动驾驶不是训练一个模型，而是在训练和管理一组不同任务、不同数据、不同指标的模型。AWS 的作用是把这些训练、评估、版本管理和部署流程组织成可追踪的工程流水线。

---

## 1. 所有模型训练都绕不开的通用语言

无论是感知模型、预测模型，还是 RL policy，训练时都会反复出现这些词：

| 术语 | 含义 | 自动驾驶例子 |
| --- | --- | --- |
| Input | 模型输入 | 摄像头图像、LiDAR 点云、车辆状态、历史轨迹 |
| Output | 模型输出 | 目标框、车道线、未来轨迹、驾驶动作 |
| Label | 人类或系统给出的正确答案 | 行人框、红灯状态、人类驾驶轨迹 |
| Loss | 训练时优化的错误度量 | 检测框误差、分类错误、轨迹预测误差 |
| Metric | 评估模型好坏的指标 | mAP、碰撞率、到达率、违规次数 |
| Dataset | 训练数据集合 | 真实驾驶日志、仿真场景、标注图片 |
| Train / Val / Test | 训练集、验证集、测试集 | 用不同城市、天气、道路切分数据 |
| Checkpoint | 训练中保存的模型快照 | 每训练一段时间保存一次到 S3 |
| Artifact | 可部署产物 | 模型文件、配置、容器镜像、评估报告 |
| Registry | 模型版本库 | SageMaker Model Registry |
| Inference | 推理 | 车端实时识别行人，或云端离线跑评估 |

AWS 对应理解：

```text
S3 存 dataset / checkpoint / artifact
SageMaker Training 跑训练
SageMaker Processing 跑离线评估
CloudWatch 看日志
Model Registry 管版本和审批
ECR 存容器镜像
EKS / Batch 跑大规模仿真和评估
```

## 2. 感知模型的训练语言

感知模型回答：

```text
周围有什么？
在哪里？
属于什么类别？
```

常见模型：

- 目标检测模型
- 语义分割模型
- 车道线检测模型
- 红绿灯识别模型
- 障碍物检测模型

训练方式通常是监督学习。

```text
输入：图像 / 点云
标签：目标框 / 分割 mask / 类别 / 车道线
输出：检测框、类别、像素类别、车道结构
损失：分类 loss + 定位 loss + 分割 loss
指标：mAP、IoU、precision、recall、漏检率、误检率
```

AWS 视角：

- 原始图像、点云放在 S3。
- 标注可以用 SageMaker Ground Truth 或外部标注系统。
- SageMaker Training 训练模型。
- SageMaker Processing 做离线评估。
- 结果和模型 artifact 回写 S3。

感知模型和其他模型的互动：

```text
感知输出 -> 融合模型 -> 预测模型 -> 规划模型
```

如果感知漏掉一个行人，后面的预测和规划可能根本不知道这个风险存在。

---

## 3. 预测模型的训练语言

预测模型回答：

```text
其他交通参与者接下来可能怎么动？
```

常见任务：

- 预测前车是否减速
- 预测旁车是否变道
- 预测行人是否横穿
- 预测未来 3-8 秒轨迹

训练方式通常是监督学习或生成式建模。

```text
输入：过去几秒的轨迹、速度、车道结构、交通灯状态
标签：未来几秒真实轨迹
输出：一条或多条可能未来轨迹
损失：轨迹误差、负对数似然、分类 loss
指标：ADE、FDE、miss rate、轨迹覆盖率
```

AWS 视角：

- 真实驾驶日志进入 S3。
- Glue / EMR / Processing 从日志中抽取轨迹样本。
- SageMaker Training 训练时序模型。
- Batch / Processing 在大量场景中评估预测稳定性。

预测模型和规划模型互动很紧：

```text
预测：左侧车 3 秒内可能加速靠近
规划：暂时不变道，保持车道或减速
```

## 4. 决策与规划模型的训练语言

决策与规划回答：

```text
我接下来应该怎么开？
应该走哪条轨迹？
```

它可能包含两层：

- 行为决策：保持车道、变道、让行、停车、汇入
- 轨迹规划：未来几秒具体走哪条路径

训练方式可能混合多种方法：

| 方法 | 学什么 | 例子 |
| --- | --- | --- |
| 规则 / 优化 | 根据明确规则和约束找可行轨迹 | 不压线、不碰撞、不超速 |
| 模仿学习 | 学人类司机怎么做 | 人类在这个场景选择让行 |
| RL | 学哪个动作带来更好长期结果 | 在仿真中学会安全变道 |
| 混合方法 | 规则兜底，学习模型优化行为 | RL 提建议，安全层做否决 |

训练语言：

```text
输入：自车状态、道路结构、感知结果、预测结果、目标路线
输出：驾驶意图或目标轨迹
监督信号：人类驾驶轨迹、规则约束、仿真奖励
指标：碰撞率、到达率、违规率、舒适度、通行效率
```

AWS 视角：

- S3 存真实驾驶轨迹和仿真场景。
- SageMaker Training 训练模仿学习或 RL 策略。
- EKS / Batch 并行跑大量仿真。
- Processing 做回放评估和指标统计。
- Model Registry 管理通过评估的策略版本。

---

## 5. 控制模型 / 控制器的训练语言

控制层回答：

```text
如何把目标轨迹变成方向盘、油门、刹车？
```

控制层不一定是深度学习模型，也可能是传统控制算法。

常见方法：

- PID
- MPC
- 车辆动力学模型
- 学习型控制
- 特定场景下的 RL 控制

训练或调参语言：

```text
输入：目标轨迹、当前位置、速度、横向误差、航向误差
输出：转向角、油门、刹车
目标：轨迹跟踪误差小，动作平滑，车辆稳定
指标：横向误差、纵向误差、jerk、急刹次数、稳定性
```

AWS 视角：

- 用仿真环境测试不同控制参数。
- 用 Processing / Batch 跑大量轨迹跟踪测试。
- 用 S3 保存控制日志和失败案例。

## 6. 安全模型和规则层的训练语言

安全层回答：

```text
这个动作能不能执行？
这个状态是否危险？
是否需要降级或接管？
```

它不一定完全靠训练，很多时候是规则、约束和验证系统。

```text
输入：规划轨迹、障碍物位置、预测轨迹、交通规则、车辆动力学边界
输出：通过 / 否决 / 降级 / 紧急制动
监督信号：事故、接管、近碰撞、违规、人工审核
指标：危险场景召回率、误报率、漏报率、响应时间
```

AWS 视角：

- S3 存接管和近碰撞日志。
- Athena / Glue / EMR 分析高风险片段。
- Processing 做离线安全回放。
- CloudTrail / IAM / Model Registry 记录模型审批和发布过程。

安全层和 RL 的关系尤其重要：

```text
RL policy 可以建议动作
安全层决定是否允许执行
控制器只执行通过安全检查的轨迹
```

## 7. 场景挖掘模型的训练语言

场景挖掘模型回答：

```text
海量驾驶日志里，哪些片段值得拿出来重新训练或评估？
```

它对数据闭环很重要。

常见目标：

- 找接管片段
- 找急刹片段
- 找低置信度片段
- 找模型和规则冲突片段
- 找罕见天气、罕见道路、罕见交通互动

训练方式：

```text
规则筛选
异常检测
聚类
监督学习
人工审核反馈
```

AWS 视角：

- Kinesis / IoT Core 接收运行日志。
- S3 存储长期日志。
- Glue / EMR / Athena 做筛选和查询。
- Processing 生成新的场景库。
- 新场景回到训练和评估流程。

## 8. RL policy 的训练语言

现在再看 RL，就更清楚它在系统中的位置了。

RL policy 回答：

```text
在当前状态下，选择哪个动作能让长期结果更好？
```

RL 基本术语：

| RL 术语 | 自动驾驶对应物 |
| --- | --- |
| Agent | 自车 |
| Environment | 道路、交通、天气、其他车、行人、规则 |
| Observation | 自车状态、车道、周围目标、预测结果 |
| Action | 加速、减速、保持车道、变道、停车 |
| Reward | 安全、到达目标、效率、舒适、守规则 |
| Penalty | 碰撞、违规、急刹、危险距离 |
| Policy | 根据状态选择动作的策略 |
| Episode | 一次完整驾驶任务 |

训练语言：

```text
输入：环境 observation
输出：action
监督信号：不是固定标签，而是 reward
优化目标：长期累计 reward 更高
指标：碰撞率、到达率、违规率、平均 reward、舒适度
```

AWS 视角：

- 仿真环境运行在 EC2 / EKS / Batch。
- RL 训练任务运行在 SageMaker Training 或 EKS。
- checkpoint 保存到 S3。
- 大规模评估用 Processing / Batch / EKS。
- 通过评估的策略进入 Model Registry。

RL 的特殊点：

> 感知和预测模型大多学习“正确答案”；RL 学的是“行动之后结果好不好”。

## 9. 不同模型的训练语言对比

| 模型类型 | 主要学习信号 | 输出 | 关键指标 |
| --- | --- | --- | --- |
| 感知模型 | 标注框、类别、mask | 周围有什么 | mAP、IoU、漏检率 |
| 预测模型 | 真实未来轨迹 | 其他人怎么动 | ADE、FDE、miss rate |
| 规划模型 | 人类轨迹、规则、reward | 自车怎么走 | 碰撞率、到达率、舒适度 |
| 控制模型 | 轨迹误差、车辆响应 | 怎么执行轨迹 | 跟踪误差、jerk、稳定性 |
| 安全模型 | 接管、事故、规则 | 是否允许执行 | 危险召回率、误报率 |
| 场景挖掘模型 | 异常、接管、低置信度 | 哪些数据值得回流 | 场景覆盖率、有效样本率 |
| RL policy | reward | 选择动作 | 长期 reward、碰撞率、违规率 |

## 10. 本阶段你需要掌握到什么程度

完成这一阶段后，你应该能说清楚：

- 感知模型主要靠标注数据训练。
- 预测模型主要靠历史轨迹和未来轨迹训练。
- 规划模型可能靠规则、模仿学习、RL 或混合方式训练。
- 控制层不一定是深度学习，也可能是传统控制。
- 安全层不是可有可无，它会约束或否决规划 / RL 输出。
- 场景挖掘模型负责把失败案例送回训练闭环。
- RL 是“用 reward 训练行动策略”的语言，不是所有模型的通用语言。
- AWS 把这些不同训练流程组织成数据、训练、评估、注册、部署、回流的闭环。

一句话总结：

> 第 2 阶段不是先问“RL 怎么训练”，而是先问“自动驾驶里有哪些模型，它们各自用什么数据、什么信号、什么指标训练”。理解了这个，RL 才能放到正确的位置。

## 11. 下一阶段预告

第 3 阶段会进入：

```text
为什么自动驾驶需要仿真
为什么真实道路不能随便试错
场景库是什么
离线回放和仿真有什么区别
训练后还需要哪些验证步骤
AWS 如何支撑大规模仿真、评估和模型审批
```

核心问题会从“模型怎么训练”推进到“模型怎么被验证到足够安全”。
