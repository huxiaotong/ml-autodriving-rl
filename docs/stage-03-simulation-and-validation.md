# 第 3 阶段：理解为什么自动驾驶需要仿真，以及其他必要步骤

目标：理解自动驾驶为什么不能只靠真实道路数据训练，也不能训练完模型就直接上车。你要看清楚仿真、离线回放、场景库、评估、安全门禁、边缘部署前验证之间的关系。

这一阶段的核心结论：

> 仿真不是为了“替代真实世界”，而是为了在可控、可重复、可扩展、低风险的环境里训练和验证自动驾驶系统。真实部署前，还必须经过离线回放、大规模场景测试、安全评估、封闭场地测试、shadow mode 和分阶段道路测试。

---

## 1. 为什么自动驾驶需要仿真

自动驾驶系统要学会处理大量复杂情况：

- 前车急刹
- 旁车强行并线
- 行人突然横穿
- 路口无保护左转
- 雨天、夜间、雾天
- 施工区车道变化
- 交通灯遮挡
- 传感器短暂失效
- 其他车辆不遵守规则

如果完全依赖真实道路训练，会遇到几个问题。

| 问题 | 为什么严重 |
| --- | --- |
| 真实试错成本高 | 车不能为了训练去撞人、撞车、违规 |
| 危险场景稀少 | 长尾场景很难等到，等到了也不能反复复现 |
| 场景不可控 | 同一个路口、天气、车流、人流很难完全复现 |
| 数据收集慢 | 要覆盖足够多城市、道路、天气、交通行为很困难 |
| RL 需要大量交互 | 强化学习需要试很多动作，真实道路无法安全探索 |
| 版本验证困难 | 新模型和旧模型必须在同一批场景上公平比较 |

所以自动驾驶需要仿真。

仿真的价值不是说“仿真一定真实”，而是：

```text
可控
可重复
可扩展
可制造危险场景
可比较不同模型版本
可安全试错
```

## 2. 仿真在训练和验证中的位置

仿真不是单独存在的，它位于整个训练闭环中间：

```text
真实驾驶日志
      |
      v
场景挖掘 / 数据清洗
      |
      v
场景库
      |
      v
仿真环境
      |
      v
训练 / 评估模型
      |
      v
离线回放 + 大规模仿真测试
      |
      v
模型审批
      |
      v
车端部署前验证
      |
      v
shadow mode / 封闭场地 / 小范围道路测试
      |
      v
日志回流
```

AWS 对应视角：

```text
S3：存真实日志、场景库、仿真结果、模型产物
Glue / EMR / Athena：筛选和分析场景
SageMaker Processing：数据处理、离线回放、评估报告
EC2 / EKS / Batch：并行运行仿真
SageMaker Training：训练模型或 RL policy
Model Registry：管理通过评估的模型版本
ECR：保存模型服务镜像
CloudWatch：收集训练、仿真、评估日志
```

## 3. 仿真环境到底模拟什么

一个自动驾驶仿真环境通常不只是画一条路。它需要模拟多个层面。

| 仿真内容 | 例子 | 为什么重要 |
| --- | --- | --- |
| 道路结构 | 车道、路口、匝道、环岛、施工区 | 决定规划和规则约束 |
| 交通规则 | 红绿灯、限速、让行、停止线 | 判断动作是否合法 |
| 其他车辆 | 跟车、变道、抢行、急刹 | 训练和测试交互决策 |
| 行人 / 骑行者 | 横穿、停顿、突然出现 | 测试安全和预测能力 |
| 车辆动力学 | 加速度、制动距离、转弯半径 | 检查控制是否可执行 |
| 传感器 | Camera、LiDAR、Radar、GPS、IMU | 测试感知和融合链路 |
| 天气 / 光照 | 夜间、雨天、雾天、逆光 | 测试感知鲁棒性 |
| 故障 / 噪声 | 传感器缺帧、定位漂移、误检漏检 | 测试系统降级能力 |

不同仿真工具侧重点不同：

| 工具 | 更适合理解什么 |
| --- | --- |
| CARLA | 3D 城市场景、传感器仿真、端到端自动驾驶实验 |
| SUMO | 大规模交通流、路网、交通行为 |
| Highway-env | 简化高速公路决策、RL 入门 |
| MetaDrive | 多场景驾驶仿真、RL 训练 |
| 自研仿真器 | 公司内部真实场景、传感器、车辆动力学、评估体系 |

学习阶段不需要一开始搭复杂 CARLA。先用简化环境理解流程，比一开始追求逼真更重要。

## 4. 场景库：仿真的核心资产

仿真的价值很大程度取决于场景库。

场景库不是随便生成一些道路，而是系统性收集和组织测试场景：

```text
正常场景
边界场景
危险场景
失败案例
接管案例
低置信度案例
规则冲突案例
```

一个场景通常包含：

| 内容 | 例子 |
| --- | --- |
| 地图 | 路口、高速、匝道、施工区 |
| 初始状态 | 自车位置、速度、目标路线 |
| 其他参与者 | 车辆、行人、骑行者的位置和行为 |
| 环境条件 | 天气、光照、路面状况 |
| 事件触发 | 前车急刹、旁车并线、行人横穿 |
| 成功标准 | 不碰撞、到达目标、不违规、舒适 |
| 失败标签 | 碰撞、急刹、压线、规划失败 |

AWS 上可以这样组织：

```text
S3 bucket
  raw-logs/
  extracted-scenarios/
  simulation-configs/
  replay-results/
  evaluation-reports/
  model-artifacts/
```

场景元数据可以放在：

- DynamoDB
- Glue Data Catalog
- Athena 可查询表
- 关系数据库
- 自研场景管理系统

核心思想：

> 自动驾驶训练和评估不是只管理模型，而是同时管理“模型版本 + 数据版本 + 场景版本 + 评估结果”。

## 5. 仿真有两种主要用法：训练和评估

### 5.1 用仿真训练

仿真训练常见于 RL 或交互式策略优化。

```text
1. 仿真环境初始化一个场景
2. 自车 policy 观察环境状态
3. policy 输出动作
4. 仿真环境更新世界
5. 系统计算 reward / penalty
6. 训练算法更新 policy
7. 重复很多 episode
```

适合训练：

- 高速变道策略
- 匝道汇入策略
- 路口交互策略
- 跟车策略
- 避障策略

AWS 对应：

```text
EKS / EC2 / Batch 并行跑多个仿真实例
SageMaker Training 或 EKS 跑训练任务
S3 保存 checkpoint、轨迹、reward、日志
CloudWatch 监控训练过程
```

### 5.2 用仿真评估

仿真评估更接近部署前测试。

```text
同一批场景
同一套评估指标
不同模型版本
横向对比表现
```

评估目标：

- 新模型是否比旧模型更安全？
- 是否修复了某类失败场景？
- 是否引入新的回归问题？
- 在长尾场景中表现如何？
- 延迟、稳定性、资源占用是否可接受？

AWS 对应：

```text
AWS Batch / EKS 并行跑评估场景
SageMaker Processing 汇总指标
S3 保存报告
Model Registry 记录是否通过审批
```

## 6. 离线回放：比仿真更贴近真实日志

离线回放 replay 是另一个必要步骤。

它不是重新创造一个世界，而是把真实道路记录下来的数据重新喂给模型：

```text
真实传感器日志
      |
      v
新模型离线推理
      |
      v
比较输出和真实结果 / 旧模型结果 / 安全规则
```

离线回放可以回答：

- 新感知模型在真实日志上是否漏检？
- 新预测模型是否更早预测出危险行为？
- 新规划模型如果当时上车，会不会做出更好决策？
- 新模型是否在旧版本通过的场景中退化？

它的好处：

- 使用真实数据
- 可重复
- 比真实道路测试安全
- 可以大规模并行
- 很适合做模型版本对比

但它也有限制：

> 离线回放不能完全评估“如果模型当时选择另一个动作，世界会如何反应”。这类反事实交互更需要仿真。

所以两者互补：

```text
离线回放：真实但不完全可交互
仿真：可交互但不完全真实
```

## 7. 为什么仿真不能替代真实道路

仿真很重要，但它有 sim-to-real gap。

差距来自：

- 传感器渲染不够真实
- 车辆动力学不完全准确
- 其他司机行为不够真实
- 行人行为难模拟
- 天气和光照难完全还原
- 真实道路有不可预测因素
- 社会性互动很复杂

因此不能说：

```text
仿真通过 = 可以直接上路
```

更正确是：

```text
仿真通过 = 可以进入下一阶段验证
```

部署前还需要：

- 离线回放
- 大规模仿真回归测试
- 安全规则检查
- 资源和延迟测试
- 硬件在环测试
- 封闭场地测试
- shadow mode
- 小范围道路测试
- 人类安全员监督

## 8. 其他必要步骤：从模型到上车

一个比较完整的验证链路如下：

```text
1. 单模型离线评估
2. 多模型接口集成测试
3. 真实日志离线回放
4. 仿真场景评估
5. 长尾场景压力测试
6. 安全层和规则检查
7. 系统延迟和资源测试
8. 硬件在环测试
9. 封闭场地测试
10. shadow mode
11. 小范围道路测试
12. 分阶段扩大部署
13. 运行日志回流
```

### 8.1 单模型离线评估

先确认每个模型自己没有明显退化。

```text
感知：mAP、漏检率、误检率
预测：ADE、FDE、miss rate
规划：碰撞率、到达率、违规率
控制：轨迹误差、jerk、稳定性
安全：危险召回率、误报率
```

### 8.2 多模型接口集成测试

检查模型之间的输入输出是否匹配。

```text
感知输出格式 -> 预测输入
预测输出格式 -> 规划输入
规划轨迹格式 -> 控制输入
安全层能否读取规划结果
```

### 8.3 真实日志离线回放

用真实数据测试新模型，但不控制真实车辆。

AWS：

```text
S3 日志 -> Processing Job -> 模型推理 -> 指标报告
```

### 8.4 大规模仿真评估

用场景库跑系统级测试。

AWS：

```text
S3 场景库 -> Batch / EKS 并行仿真 -> S3 结果 -> Processing 汇总
```

### 8.5 硬件在环测试

把真实车载计算设备接入测试系统，看模型在目标硬件上是否满足：

- 延迟
- 吞吐
- GPU / CPU / 内存占用
- 温度和功耗
- 进程重启和降级策略

### 8.6 Shadow mode

模型上车但不控制车辆，只在后台“提出建议”。

```text
人类或旧系统实际开车
新模型只记录它会怎么做
系统比较新模型建议和真实行为
```

shadow mode 可以发现：

- 新模型是否经常想做危险动作
- 它和当前系统在哪些场景分歧大
- 它是否在真实环境中稳定运行

### 8.7 分阶段道路测试

道路测试必须逐步扩大：

```text
封闭场地
      |
      v
低速园区
      |
      v
固定路线
      |
      v
限定城市 / 限定天气 / 限定时段
      |
      v
更大范围
```

        v
EC2 / EKS / AWS Batch

## 10. 本阶段你需要掌握到什么程度

完成本阶段后，你应该能解释：

- 自动驾驶需要仿真，因为真实道路不能安全、大规模、可重复地试错。
- 仿真既用于训练，也用于部署前评估。
- 场景库是仿真的核心资产。
- 离线回放使用真实日志，但不能完全模拟反事实交互。
- 仿真可交互，但和真实世界有 sim-to-real gap。
- 模型不能通过单项指标就上车，必须经过系统级验证。
- AWS 可以支撑数据湖、场景库、并行仿真、训练、评估、模型审批和日志回流。

一句话总结：

> 自动驾驶的仿真不是一个游戏环境，而是训练、评估、回归测试和安全审批体系的一部分。真正的目标不是“仿真里跑得好看”，而是通过仿真和真实日志构成的验证链路，尽可能早地发现系统失败模式。

## 11. 下一阶段预告

第 4 阶段会进入一个最小训练流程：

```text
定义驾驶任务
定义仿真环境
定义 observation
定义 action
定义 reward
运行训练循环
保存 checkpoint
离线评估
映射到 AWS 训练流水线
```

核心问题会从“为什么需要仿真和验证”推进到“一个策略模型到底是怎么被训练出来的”。
